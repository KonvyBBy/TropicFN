COSMETIC IMAGE FETCHING — CURRENT METHOD vs. FASTER METHOD
===========================================================

=== CURRENT METHOD ===

File: web_app.py  (function: fortnite_api_get_cosmetic_icon_url_by_name, line ~43)
Endpoint used: https://fortnite-api.com/v2/cosmetics/br/search/all

How it works right now:
1. When the dashboard or an API route needs the icon for a cosmetic (skin, emote, pickaxe, glider),
   it calls fortnite_api_get_cosmetic_icon_url_by_name(name, type).
2. That function first checks a local file cache (cosmetic_icon_cache.json) to see if we already
   looked up this name before. If yes, it returns the cached URL instantly.
3. If the name is NOT in the cache, it fires a SEPARATE HTTP GET request to fortnite-api.com,
   passing the cosmetic name as a query parameter (?name=<name>&matchMethod=contains).
4. The response is parsed and the first matching image URL is saved to the cache file and returned.
5. The /api/skins/icons endpoint loops through every name in the request one-by-one, so if you ask
   for 50 skins it makes up to 50 separate HTTP requests, one after another (sequential).

Why this is slow:
- Every unknown cosmetic = 1 individual outbound HTTP call (~0.5 – 3 seconds each, depending on
  network/server load).
- Requests are sequential (one finishes before the next starts), so a typical account with
  100 skins + 30 emotes + 10 pickaxes = up to 140 sequential HTTP calls.
- Worst-case total wait time: 140 calls × 2 seconds = ~280 seconds (~4-5 minutes) for a cold cache.
- The cache file (cosmetic_icon_cache.json) helps on repeat lookups for the same names, but every
  new/unique cosmetic name still triggers a slow individual fetch.


=== NEW (FASTER) METHOD ===

Use the bulk "all cosmetics" endpoint and build a full in-memory lookup table at app startup.

Endpoint: https://fortnite-api.com/v2/cosmetics/br
(returns ALL Battle Royale cosmetics in a single response — several thousand items)

How it would work:
1. At app startup (or once per day via a background scheduler), make ONE single HTTP request to
   https://fortnite-api.com/v2/cosmetics/br to download the full cosmetics database.
2. Parse the response and build a Python dictionary keyed by cosmetic name (lowercase) mapping to
   its icon URL. Store this dict in memory (and optionally persist it to a JSON file so it survives
   restarts without a re-fetch).
3. When any code needs an icon, it does a simple dict lookup (O(1), sub-millisecond) — no HTTP
   call, no file I/O.
4. Schedule a refresh (e.g., every 24 hours) to pick up newly-added cosmetics.

Example pseudocode:
    # At startup
    resp = requests.get("https://fortnite-api.com/v2/cosmetics/br")
    all_cosmetics = resp.json()["data"]   # list of thousands of items
    COSMETIC_LOOKUP = {}
    for item in all_cosmetics:
        name_key = item["name"].lower()
        images = item.get("images") or {}
        url = images.get("icon") or images.get("smallIcon") or images.get("featured")
        COSMETIC_LOOKUP[name_key] = url

    # For every lookup afterwards
    def get_icon(name):
        return COSMETIC_LOOKUP.get(name.lower())


=== SPEED COMPARISON ===

Scenario: Displaying icons for an account with 100 skins, 30 emotes, 10 pickaxes, 5 gliders
          (145 total cosmetics, all cold cache / first time seen)

Current method (sequential per-name requests):
  - 145 HTTP calls × ~1-2 sec each = 145 – 290 seconds total
  - User stares at a loading spinner for 2–5 minutes

New method (single bulk fetch + in-memory dict):
  - 1 HTTP call at startup = ~1-2 seconds (done ONCE, not per request)
  - 145 dict lookups = ~0.0001 seconds total
  - User sees icons almost instantly (<1 ms for the lookup itself)

Speed improvement for lookup phase: ~1,000x – 100,000x faster
Overall user-facing improvement: from minutes to near-instant


=== ADDITIONAL BENEFITS OF THE NEW METHOD ===

- No more per-request rate-limiting risk on fortnite-api.com (one bulk call vs. hundreds).
- The in-memory dict is shared across all requests, so all users benefit simultaneously.
- Easier to cache to disk: dump the whole dict to a JSON file once a day; load it at startup.
- Works offline once the bulk file is cached locally.
- The cosmetics database rarely changes mid-day, so a 24-hour refresh cycle is more than sufficient.
